# CloudSec Terraform Project

A production-grade WordPress deployment on AWS, fully automated with Terraform. This project provisions a secure, scalable, and highly available infrastructure following the **AWS Well-Architected Framework** — with zero hardcoded credentials, private compute, and full observability.

---

## Architecture Overview

```
Internet
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  Route 53 (DNS)  →  ACM Certificate (SSL/TLS)           │
└─────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  Application Load Balancer  (public subnets)            │
│  • HTTP :80  →  301 redirect to HTTPS                   │
│  • HTTPS :443  →  forwards to ECS target group          │
└─────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  ECS Fargate Tasks  (PRIVATE subnets — not public)      │
│  • WordPress container (php8.3-apache)                  │
│  • Secrets fetched from SSM at startup                  │
│  • Mounts EFS for shared wp-content storage             │
│  • Outbound via NAT Gateway (no public IP on tasks)     │
└──────────────┬──────────────────────┬───────────────────┘
               │                      │
               ▼                      ▼
┌──────────────────────┐   ┌──────────────────────────────┐
│  RDS MySQL 8.0       │   │  EFS (Elastic File System)   │
│  (private subnets)   │   │  (private subnets)           │
│  • Multi-AZ standby  │   │  • Encrypted at rest         │
│  • Automated backups │   │  • Encrypted in transit      │
│  • Not public        │   │  • AWS Backup enabled        │
└──────────────────────┘   └──────────────────────────────┘
```

| Layer | Service | Why |
|---|---|---|
| Compute | ECS Fargate | Serverless containers — no EC2 to patch or manage |
| Database | RDS MySQL 8.0 | Managed, Multi-AZ, automated backups |
| Storage | EFS | Shared persistent storage across all ECS tasks |
| Networking | VPC + NAT Gateway | Private compute with controlled outbound access |
| Load Balancing | Application Load Balancer | SSL termination, health checks, traffic distribution |
| SSL/TLS | ACM | Free, auto-renewing certificates |
| DNS | Route 53 | Alias record pointing domain to ALB |
| Secrets | SSM Parameter Store | Encrypted credentials, never in environment variables |
| Observability | CloudWatch + SNS | Alarms with email notifications |
| Scaling | ECS Auto Scaling | CPU and memory-based scale-out |
| CI/CD | GitHub Actions (OIDC) | No long-lived AWS keys stored anywhere |

## Security Design

### No hardcoded credentials
The DB password is randomly generated by Terraform (`random_password`) and stored in SSM Parameter Store as a `SecureString` (KMS-encrypted). The username is also stored as `SecureString`. Neither value ever appears in plaintext in your code, state file display, or environment variables.

### Secrets injected at runtime
ECS fetches credentials directly from SSM at task startup using the `secrets` field in the task definition. The values are injected as environment variables inside the container only — they are never written to disk or logged.

### ECS tasks are NOT publicly accessible
Tasks run in **private subnets** with `assign_public_ip = false`. The only way to reach them is through the ALB. Outbound traffic (SSM API calls, DockerHub image pulls) goes through a NAT Gateway — tasks never have a public IP.

### Least-privilege security groups
Every security group only allows the minimum traffic required:

| Group | Accepts inbound from | Sends outbound to |
|---|---|---|
| ALB | Internet (80, 443) | ECS tasks (80) only |
| ECS | ALB (80) only | RDS (3306), EFS (2049), internet HTTPS (443 via NAT) |
| RDS | ECS (3306) only | Nothing |
| EFS | ECS (2049) only | Nothing |

### Least-privilege IAM
- The **execution role** (used by the ECS agent) can only read the two specific SSM parameters this service needs — not all SSM parameters in the account.
- The **task role** (used by the WordPress container) can only access the specific EFS file system it mounts — not all EFS resources.
- The **GitHub Actions role** has a scoped policy covering only the AWS services Terraform needs to manage — not `Action: * Resource: *`.

### HTTPS enforced end-to-end
HTTP (port 80) returns a 301 permanent redirect to HTTPS. SSL terminates at the ALB using a TLS 1.3-compatible security policy (`ELBSecurityPolicy-TLS13-1-2-2021-06`). Traffic between the ALB and ECS is on port 80 inside the VPC (private network).

### EFS encrypted at rest and in transit
`encrypted = true` on the file system. `transit_encryption = ENABLED` on the ECS volume mount. IAM authorization is enforced on the access point.

### VPC Flow Logs
All IP traffic in and out of the VPC is captured to CloudWatch Logs for security auditing and incident investigation.

## Reliability Design

### Multi-AZ everything
- **ECS**: `min_capacity = 2` ensures at least one task per AZ. If an AZ goes down, the other task keeps serving traffic.
- **RDS**: `multi_az = true` (for production) creates a synchronous standby replica in a second AZ. Failover is automatic with no data loss.
- **EFS**: Mount targets in both private subnets — tasks in either AZ can mount the same file system.
- **ALB**: Spans both public subnets across both AZs.

### RDS data protection
- `deletion_protection = true` — prevents accidental deletion via Terraform or the console.
- `skip_final_snapshot = false` — takes a named snapshot before any destroy operation.
- `backup_retention_period = 7` — 7 days of automated daily backups for point-in-time recovery.
- `max_allocated_storage = 100` — storage autoscales up to 100 GB automatically.

### ALB health checks
Health check path is `/wp-login.php` with matcher `200,302`. This works both before and after WordPress setup (unlike `/wp-admin/install.php` which redirects post-setup and would cause false health check failures).

### EFS backups
`aws_efs_backup_policy` with `status = ENABLED` activates AWS Backup for the EFS file system. Without this, EFS data is not backed up automatically.

## Observability

### CloudWatch Alarms (all wired to SNS)

| Alarm | Condition | What to do |
|---|---|---|
| `ecs-cpu-high` | ECS CPU > 85% for 2 min | Check for traffic spike; auto scaling should handle it |
| `ecs-memory-high` | ECS memory > 85% for 2 min | Check for memory leaks; consider increasing task memory |
| `rds-storage-low` | RDS free storage < 5 GB | Storage autoscaling should expand; investigate if alarm persists |
| `rds-cpu-high` | RDS CPU > 80% for 3 min | Consider upgrading instance class or adding read replicas |
| `rds-connections-high` | DB connections > 50 | Consider adding RDS Proxy to pool connections |
| `alb-5xx-high` | ALB 5xx errors > 10/min | Application is returning errors; check ECS task logs |

### SNS Notifications
All alarms send to an SNS topic. Set `alert_email` in your `terraform.tfvars` to subscribe your email address. You can also subscribe Slack, PagerDuty, or any webhook to the SNS topic ARN output after deployment.

### VPC Flow Logs
Captured to `/vpc/cloudsec-vpc/flow-logs` in CloudWatch Logs. Useful for debugging connectivity issues and security investigations.

### ECS Container Logs
All container stdout/stderr goes to `/ecs/cloudsec-cluster` in CloudWatch Logs with 30-day retention (configurable via `log_retention_days`).

## Cost Optimization

### FARGATE_SPOT
The ECS cluster is configured with both `FARGATE` and `FARGATE_SPOT` capacity providers. The default strategy uses Spot for most tasks (up to 70% cheaper) with at least one on-demand task guaranteed (`base = 1` on FARGATE). For production, adjust the weights in `modules/ecs/main.tf`.

### Single NAT Gateway for dev
`nat_gateway_count = 1` by default — one NAT Gateway shared across both AZs. For production HA, set `nat_gateway_count = 2` to have one per AZ (eliminates cross-AZ NAT traffic charges and removes the NAT as a single point of failure).

### RDS storage autoscaling
`max_allocated_storage = 100` lets RDS expand storage automatically. You pay only for what you use, and you avoid manual intervention when storage fills up.

## Prerequisites

Before deploying, you need:

- An [AWS account](https://aws.amazon.com/)
- [Terraform](https://developer.hashicorp.com/terraform/install) >= 1.6.6 installed locally
- An S3 bucket for Terraform remote state (create your own — must be globally unique)
- A DynamoDB table for state locking (see step 3 below)
- A registered domain name managed in Route 53
- An ACM certificate issued for your domain in the **same region** as your deployment

## Deployment Guide

### Step 1 — Clone the repository

```bash
git clone https://github.com/<your-username>/CloudSec-Terraform-Project.git
cd CloudSec-Terraform-Project
```

### Step 2 — Create the S3 bucket and DynamoDB lock table

These must exist before running `terraform init`. Create them once manually:

```bash
# Create the S3 bucket for remote state
aws s3api create-bucket \
  --bucket <your-unique-bucket-name> \
  --region us-east-2 \
  --create-bucket-configuration LocationConstraint=us-east-2

# Enable versioning so you can recover from accidental state corruption
aws s3api put-bucket-versioning \
  --bucket <your-unique-bucket-name> \
  --versioning-configuration Status=Enabled

# Create the DynamoDB table for state locking
# This prevents two pipeline runs from corrupting state simultaneously
aws dynamodb create-table \
  --table-name cloudsec-terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-2
```

### Step 3 — Update the backend configuration

In `source/provider.tf`, update the bucket name:

```hcl
backend "s3" {
  bucket         = "<your-unique-bucket-name>"
  key            = "infra/terraform.tfstate"
  region         = "us-east-2"
  dynamodb_table = "cloudsec-terraform-locks"
  encrypt        = true
}
```

Do the same in `openid/provider.tf` (change the key to `"openid/terraform.tfstate"`).

### Step 4 — Configure your variables

```bash
cd source
cp terraform.tfvars.example terraform.tfvars
```

Open `terraform.tfvars` and fill in your values:

```hcl
# Required
certificate_arn = "arn:aws:acm:us-east-2:<account-id>:certificate/<cert-id>"
hosted_zone_id  = "<your-route53-hosted-zone-id>"
domain_name     = "yourdomain.com"
alert_email     = "you@example.com"

# Environment
environment = "dev"   # Change to "prod" for production deployments

# For production — uncomment and set these:
# rds_multi_az        = true
# nat_gateway_count   = 2
# rds_deletion_protection = true
```

### Step 5 — Deploy

```bash
terraform init
terraform validate
terraform plan
terraform apply
```

After apply completes, Terraform outputs:

```
wordpress_url        = "https://yourdomain.com"
alb_dns_name         = "cloudsec-alb-xxxx.us-east-2.elb.amazonaws.com"
rds_endpoint         = "cloudsec-rds.xxxx.us-east-2.rds.amazonaws.com:3306"
ecs_cluster_name     = "cloudsec-cluster"
ecs_service_name     = "cloudsec-cluster-service"
ssm_db_password_path = "/cloudsec/us-east-2/rds/password"
ssm_db_username_path = "/cloudsec/us-east-2/rds/username"
sns_alerts_topic_arn = "arn:aws:sns:us-east-2:xxxx:cloudsec-alerts"
vpc_id               = "vpc-xxxx"
```

Browse to your `wordpress_url` to complete the WordPress setup wizard.

> **Note:** If you set `alert_email`, check your inbox and confirm the SNS subscription to start receiving alarm notifications.

### Step 6 — Destroy (when needed)

```bash
terraform destroy
```

> **Warning:** If `rds_deletion_protection = true` (recommended for production), you must first set it to `false` and apply before destroy will succeed.

## CI/CD with GitHub Actions

Two workflows are provided. Both follow the same pattern: **plan → review → apply**, with destroy requiring a separate approval gate.

### How the workflow works

```
workflow_dispatch (plan / apply / destroy)
        │
        ▼
   ┌─────────┐
   │  Plan   │  Always runs. Saves plan as artifact.
   └────┬────┘
        │  (if action = apply)
        ▼
   ┌─────────┐
   │ Approve │  GitHub Environment protection rule — requires a reviewer.
   └────┬────┘
        │
        ▼
   ┌─────────┐
   │  Apply  │  Uses the saved plan artifact — applies exactly what was reviewed.
   └─────────┘

   (if action = destroy)
        ▼
   ┌──────────────┐
   │ Approve      │  Separate "destroy-approval" environment — requires a reviewer.
   └──────┬───────┘
          ▼
   ┌──────────────┐
   │   Destroy    │
   └──────────────┘
```

### Why save the plan as an artifact?
Without this, the apply step runs a fresh `terraform plan` internally. Between your review and the apply, infrastructure could have changed — meaning you apply something different from what you approved. Saving the plan file ensures apply executes exactly what you reviewed.

### Option A — Access Keys

Uses `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` stored as GitHub repository secrets.

**Required secrets:**

| Secret | Description |
|---|---|
| `AWS_ACCESS_KEY_ID` | IAM user access key |
| `AWS_SECRET_ACCESS_KEY` | IAM user secret key |

Workflow file: `.github/workflows/action.yml`

### Option B — OIDC (recommended)

Uses OpenID Connect to assume an IAM role directly from GitHub Actions. No AWS keys stored anywhere — GitHub gets a short-lived token per workflow run.

**Step 1:** Provision the OIDC IAM role (one-time setup):

```bash
cd openid
```

Edit `openid.tf` and replace `"repo:<your-github-username>/*"` with your actual GitHub username.

```bash
terraform init
terraform apply
```

This outputs the role ARN.

**Step 2:** Add the role ARN as a GitHub secret:

| Secret | Description |
|---|---|
| `AWS_ROLE_ARN` | ARN of the GithubActions-CloudSec role created above |

Workflow file: `.github/workflows/oidc.yaml`

### Setting up approval gates

In your GitHub repository, go to **Settings → Environments** and create two environments:

- `production` — add required reviewers for apply
- `destroy-approval` — add required reviewers for destroy (recommend requiring 2 reviewers)

When a workflow reaches these environments, GitHub pauses and waits for a reviewer to approve before continuing.

## Infrastructure Details

### Networking

| Resource | Count | Details |
|---|---|---|
| VPC | 1 | `10.0.0.0/16`, DNS hostnames enabled |
| Public Subnets | 2 | `10.0.1.0/24`, `10.0.3.0/24` — ALB lives here |
| Private Subnets | 2 | `10.0.2.0/24`, `10.0.4.0/24` — ECS, RDS, EFS live here |
| Internet Gateway | 1 | Attached to public route table |
| NAT Gateway | 1 (dev) / 2 (prod) | Allows private resources to reach the internet outbound |
| Route Tables | 3 | 1 public + 1 per private subnet (each pointing to its NAT) |
| VPC Flow Logs | 1 | Captures all VPC traffic to CloudWatch Logs |

### ECS Auto Scaling

| Policy | Metric | Target | Scale-out cooldown | Scale-in cooldown |
|---|---|---|---|---|
| CPU | ECSServiceAverageCPUUtilization | 70% | 60s | 300s |
| Memory | ECSServiceAverageMemoryUtilization | 80% | 60s | 300s |

Min tasks: **2** (one per AZ) — Max tasks: **4**

# Scale up for production
instance_class    = "db.t3.small"

ecs_max_capacity  = 8

## Well-Architected Framework Alignment

| Pillar | What this project does |
|---|---|
| **Operational Excellence** | Modular Terraform, environment variable, CI/CD with plan artifacts and approval gates, CloudWatch alarms with SNS notifications |
